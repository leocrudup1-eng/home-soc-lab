### Alert 1 — Wazuh Rootcheck: chfn Trojaned
**Tool:** Wazuh rootcheck  
**Rule ID:** 510 | **Level:** 7  
**Source:** Raspberry Pi — `/usr/bin/chfn`

**What happened:**
Wazuh rootcheck flagged `/usr/bin/chfn` (change finger — standard Linux user info utility) as potentially trojaned using a generic pattern match.

**Investigation:**
```bash
dpkg -V passwd        # verify package integrity
stat /usr/bin/chfn    # check file metadata
file /usr/bin/chfn    # verify file type
ls -la /usr/bin/chfn  # check permissions
```
All checks returned expected results. Binary was unmodified, signed, correct permissions.

**Verdict:** ❌ False Positive — rootcheck generic signature, binary is clean

**Action:** No suppression. Rootcheck must remain active for real compromise detection.

### Alert 8 — SURICATA SMB Malformed Request
**Tool:** Suricata  
**Rule ID:** 2225005 | **Priority:** 3  
**Source:** 192.168.x.10:port → 192.168.x.12:445

68 seconds after Alert 7.

**What happened:**
Malformed SMB dialect negotiation detected from the same source as Alert 7.

**Investigation:**
nmap's `-sV`/`-A` service version detection sends non-standard SMB dialect requests that Suricata flags as malformed. Same source IP as Alert 7 confirmed this is the same scan.

**Attack pattern identified:**
```

    ↓ 68 seconds
Alert 8: malformed SMB probe port 445   (T1046 - same scan, SMB target)
```
This is a recognizable recon pattern. In production: immediate investigation.

**Real world significance:**
SMB port 445 is historically the most exploited service in enterprise networks (EternalBlue, WannaCry, NotPetya). Malformed SMB from internal host is high priority.

**Verdict:** ✅ True Positive — authorized lab scan

**Action:** Documented. No suppression.

## Key Lessons Learned This Session

**1. Context is everything**
The Tor alert was a True Positive — but authorized. Without context (was Tor intentionally running?) the verdict would be wrong. Always identify intent before classifying.

**2. Suppress precisely, not broadly**
Tor was suppressed only for `127.0.0.1`, not globally. This is correct — unexpected Tor from any other host still alerts.

**3. ET INFO ≠ threat**
ET INFO rules are visibility tools. They tell you what's on your network, not that something is wrong.

**4. SPAN port noise is structural**
Stream errors from a mirror port are expected. Suppress them early so they don't drown real alerts.

**5. Correlation makes alerts meaningful**
Alert 7 alone is notable. Alert 7 + Alert 8 from the same IP 68 seconds apart is a recognizable recon pattern — much more actionable than either alert alone.

**6. Use multiple verification tools**
The rootcheck investigation used `dpkg`, `stat`, `file`, and `ls` independently. No single tool gives the full picture. Confidence comes from multiple corroborating sources.
